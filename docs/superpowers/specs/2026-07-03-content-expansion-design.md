# Prompt Golf 콘텐츠 확장 설계 — 도메인 네이티브 Judge & 홀 콘텐츠 팩토리

- 작성일: 2026-07-03
- 상태: 검토 대기 (기획·설계 단계, 구현 아님)
- 범위: 이미지 재현을 넘어 프론트엔드/백엔드/인프라·SRE/비개발 직군별 결과물을 측정하는 홀(hole) 콘텐츠 체계와 자동 제작 파이프라인의 설계

## 1. 배경과 목표

현재 Prompt Golf는 "목표 이미지 → 프롬프트 → Gemini HTML 생성 → 캡처 → 시각 유사도 = 비거리"라는
단일 트랙 게임이다. 이를 **사내 이벤트/팀빌딩** 맥락에서 직군별(FE/BE/인프라·SRE/비개발) 결과물로
확장한다.

핵심 결정:

- **안 B — 도메인별 네이티브 Judge 채택.** 직군마다 그 직군다운 측정(접근성 감사, SQL 실행 결과 비교,
  manifest 검증, 장애 진단 매칭)을 붙인다. "오, 진짜 우리 일이네"라는 감탄과 교육 효과를 우선한다.
- **프로젝트의 무게중심은 게임 클라이언트가 아니라 콘텐츠.** 홀 하나는 미니 eval이고, 이 문서가
  설계하는 것은 사실상 "eval 팩토리"다. LLM 평가셋 품질 기법(골든 솔루션, 변별력 검증, 난이도
  캘리브레이션)을 홀 제작에 그대로 적용한다.
- 콘텐츠는 **자동 생성 파이프라인 + 사람 최종 검수** 체제로 생산한다.

## 2. 핵심 추상화 — 홀은 4-튜플이다

게임의 본질은 이미지 게임이 아니라 **재현 챌린지(reproduction challenge)** 다. 모든 홀은 아래
4개 요소로 기술되며, 도메인 확장 = 이 튜플의 새 조합 정의다.

| 요소 | 역할 | 현행 구현 |
|---|---|---|
| **Target** | 플레이어에게 제시되는 재현/해결 목표 | 목표 이미지 |
| **Generator** | 프롬프트 → 아티팩트 생성 | Gemini HTML 생성 |
| **Renderer** | 아티팩트를 관측 가능하게 만듦 | iframe + html2canvas |
| **Judge** | 0~1 스칼라 점수 산출 → 비거리 | Gemini 비전 유사도 |

Judge가 지켜야 할 게임 성립 조건:

1. **스칼라**: 0~1 점수 하나로 수렴한다 (기존 `calculateShot()` 인터페이스 유지).
2. **공정성**: 같은 아티팩트에는 같은 점수. 결정적 채점을 최우선, LLM-judge는 보조.
3. **속도**: 스윙당 채점 15초 이내.

측정 방식은 3계열로 분류되며, 우선순위는 ① > ② > ③이다.

1. **결정적 채점** — 테스트 통과율, 감사 도구, 구조 diff. 반박 불가, 경쟁에 최적.
2. **지각적 유사도** — 이미지/프레임 비교, 임베딩 거리. 현행 방식.
3. **LLM-judge 루브릭** — 유연하나 변동성 있음. 결정적 채점이 불가능한 곳(진단문 평가 등)에만 사용.

## 3. 세션·게임 형태

- **개인전.** 플레이어 1명이 공 1개를 도맡는다. 1샷 = 1스윙(프롬프트 1회). 전원이 동시에 같은
  코스를 자기 페이스로 진행하고, 리더보드는 총 타수 기준. 현행 REST 멀티플레이어 + 관전 구조를
  그대로 사용한다.
- **세션 길이 20~30분.** 스윙 사이클 ≈ 90초(작성 60s + 생성 15s + 채점 15s) 기준으로
  **4~5홀 × 평균 파 3~4** 편성. 플레이어당 스윙 예산 15~16회.
- **파(par)가 난이도·형평성 장치.** 직군 전문성이 필요한 홀은 파를 높게 준다. 혼합 직군이 같은
  코스를 돌아도 골프 메타포가 난이도 차를 흡수한다.
- **트래픽·비용**: 스윙 트래픽 = 동시접속 × 홀당 타수. 홀 메타데이터에 홀당 평균 토큰 비용을
  기록해, 코스 편성 시 운영자가 총비용과 rate limit을 사전 계산할 수 있게 한다.

## 4. Hole Manifest — 홀은 선언적 데이터다

홀은 코드가 아니라 매니페스트(YAML/JSON)로 정의한다. 게임 코어는 매니페스트를 해석해
Generator/Renderer/Judge를 조립하는 플러그인 구조가 된다.

```yaml
hole:
  id: be-erd-blog-001
  domain: backend            # fe | be | infra | sre | nondev
  par: 4
  title: "블로그 서비스 스키마"
  brief: "플레이어에게 보여줄 목표 설명"
  cost_estimate_tokens: 12000   # 코스 편성용 비용 메타데이터
  target:                    # 플레이어가 보는 것
    type: image | text | logs
    assets: [...]
  generator:                 # 프롬프트가 만들 아티팩트
    artifact: html | sql | openapi | mermaid | k8s-manifest | config-fix | diagnosis | image
    system_prompt: "..."     # 아티팩트 타입별 생성 가드레일
  judge:                     # 가중 합성 채점 파이프라인
    scorers:
      - type: sql-result-diff
        weight: 0.7
        assets: { schema, seed, hidden_queries }
      - type: visual-similarity
        weight: 0.3
  qa:                        # 파이프라인이 채워넣는 품질 메타데이터
    golden_solution: "..."        # 만점 근처(≥0.85)가 나와야 하는 정답
    adversarial_solution: "..."   # 낮은 점수(≤0.4)가 나와야 하는 함정 답
    calibration: { weak: 0.3, mid: 0.6, strong: 0.9 }
```

설계 결정:

- **Judge는 가중 합성 파이프라인이다.** 도메인 네이티브 채점(정확)과 시각 유사도(직관·관전 재미)를
  섞을 수 있다. 가중치 0의 scorer는 점수에 무관하게 화면 표시용(관전)으로만 쓴다.
- **`qa` 블록은 일급 시민이다.** 골든/함정 솔루션이 홀 데이터에 내장되어야 자동 QA 게이트와
  회귀 검증이 성립한다.
- 유효성 게이트 실패(파싱 불가 등)는 현행 "유사도 < 0.3 → 헛스윙" 규칙과 동형으로 처리한다.

## 5. 도메인별 홀·Judge 설계

### 5.1 채점 실행 전략 — 대부분 브라우저 안에서

안 B의 최대 리스크(채점용 실행 인프라)는 WASM/JS 클라이언트 채점으로 대부분 제거한다.

| Judge | 실행 방법 | 인프라 |
|---|---|---|
| SQL 실행·결과 비교 | sql.js(SQLite WASM) 또는 PGlite(Postgres WASM) | 없음 (클라이언트) |
| 접근성 감사 | axe-core — 기존 캡처용 iframe에서 실행 | 없음 |
| OpenAPI 검증·diff | swagger-parser + ajv | 없음 |
| K8s manifest 검증 | JS 스키마 검증(kubeconform식) + 구조 diff | 없음 |
| 정책 검사 | OPA(Rego) WASM 빌드 | 없음 |
| 설정 파일 룰 체크 | nginx/YAML/INI 파서(JS) + 룰 평가 | 없음 |
| Lighthouse, Playwright 시나리오 | headless Chrome | 서버 필요 → 선택 홀로 격리 |
| terraform plan | 실 provider 필요 | HCL 구조 diff로 대체 (비목표 §8) |

### 5.2 도메인별 상세

| 도메인 | 홀 예시 | 플레이어가 보는 것 | 생성 아티팩트 | 네이티브 Judge |
|---|---|---|---|---|
| **FE** | "이 페이지를 재현 + 키보드 조작 가능해야 함" | 목표 스크린샷(+동적이면 GIF/다중 프레임) | HTML/CSS/JS | 시각 유사도 + axe 통과율 + (선택) Playwright 시나리오 |
| **BE/DB** | "요구사항의 스키마 설계", "이 결과가 나오는 쿼리" | 요구사항 카드 or 기대 결과 테이블 | SQL DDL/쿼리 | sql.js 실행 → 히든 쿼리 결과 diff, 스키마 구조 매칭 F1 |
| **BE/API** | "이 요구사항의 API 명세 작성" | 요구사항 카드 + 예시 요청/응답 2~3개 | OpenAPI 3.x YAML | §5.3 4층 채점 |
| **인프라** | "이 아키텍처를 K8s manifest로" | 아키텍처 다이어그램 | YAML manifest | 스키마 검증 + 구조 diff + OPA 정책 통과율 |
| **SRE** | 장애 진단/수정 (§5.4 2티어) | 장애 로그/메트릭 그래프 | 진단문 / 설정 수정본 | 임베딩+키워드 루브릭 / 증상 해소 룰셋 |
| **비개발** | "이 포스터를 재현하라" | 목표 이미지 | 이미지 생성(기존 OpenAI 트랙) | 시각 유사도(현행) |

FE 동적/애니메이션 홀: 목표를 GIF 또는 t=0/0.5s/1s 프레임 시퀀스로 제시하고, 캡처도 동일 시점
프레임 시퀀스로 비교한다.

### 5.3 API 명세 홀 — 4층 채점 (전부 클라이언트 JS)

| 층 | 내용 | 방식 | 가중치(안) |
|---|---|---|---|
| 유효성 게이트 | 파싱 + OpenAPI 3.x 스키마 검증 실패 → 헛스윙 | swagger-parser | 게이트 |
| 엔드포인트 매칭 | 히든 정답 스펙과 (path, method) 쌍 매칭 F1. path 파라미터 이름 정규화(`/users/{id}` ≡ `/users/{userId}`) | 구조 diff | 0.4 |
| 스키마 충실도 | 매칭된 엔드포인트별 요청 파라미터·응답 필드의 타입 인지 매칭 F1. 필드명 camel/snake 정규화 + 임베딩 보조 매칭(`created_at` ≈ `createdAt`) | 구조 diff | 0.3 |
| 히든 계약 테스트 | 내장된 요청/응답 예시 케이스가 생성 스펙으로 검증 통과하는 비율. 에러 계약(404, 422 등)을 히든 케이스에 섞어 "행복 경로만 명세하면 파온 불가"를 게임 규칙화 | ajv | 0.3 |

관전용 보조: Swagger UI 렌더 캡처의 시각 비교를 가중치 0으로 화면에만 표시.

### 5.4 SRE — 난이도 2티어

- **티어 1 (파 3) — 진단 홀**: 로그·메트릭 제시 → 자연어 진단문 → 정답 원인과 임베딩 유사도 +
  필수 키워드 루브릭(예: "커넥션 풀"과 "고갈" 개념이 모두 있어야 만점권).
- **티어 2 (파 5) — 진단+수정 홀**: 2단계. 1단계는 티어 1과 동일한 진단, 2단계는 수정 프롬프트로
  깨진 설정/쿼리를 고친다. **라이브 시뮬레이션 대신 결정적 체커**를 쓴다: 홀 자산에 깨진 설정 +
  골든 수정본 + **증상 해소 룰셋**(예: "nginx 설정 파싱 결과 `proxy_read_timeout ≥ 60` 이고
  upstream에 `keepalive` 존재 시 해소 판정")을 내장하고, JS 파서 + 룰 평가로 채점한다.
  홀 제작 비용이 "룰셋 작성"으로 줄고, 채점은 완전 결정적이다.
- 티어 2는 코스의 **시그니처 홀**(마지막 홀 배치)로 쓴다.

## 6. 콘텐츠 자동 생산 파이프라인 — 5단계

사람 최종 검수를 전제로 한다.

```
① 시드 브리프 → ② 홀 생성(LLM) → ③ 자동 QA 게이트 → ④ 사람 검수 콘솔 → ⑤ 홀 라이브러리
```

1. **시드 브리프**: 운영자가 `도메인 × 테마 × 목표 난이도`만 입력 (예: `be × 커머스 × par4`).
   이벤트마다 회사 도메인 테마를 입히는 지점.
2. **홀 생성 (LLM)**: 매니페스트 전체를 생성 — 목표물, judge 자산(히든 쿼리, 시드 데이터,
   정책 룰, 증상 해소 룰셋), 그리고 **골든 솔루션과 함정 솔루션까지 함께** 생성.
3. **자동 QA 게이트** — 파이프라인의 심장. 하나라도 실패하면 사람에게 가지 않는다.
   - **풀이 가능성**: 골든 솔루션 점수 ≥ 0.85
   - **변별력**: 함정/무성의 솔루션 점수 ≤ 0.4 ("아무거나 내도 맞는 홀" 차단)
   - **결정성**: 같은 아티팩트 3회 채점 시 표준편차 ≤ 0.05 (LLM-judge 포함 홀에서 특히 중요)
   - **난이도 캘리브레이션**: 약/중/강 3단계 프롬프트 실력의 시뮬레이션 플레이어가 실제 플레이
     → 점수 분포로 par 자동 보정. "강 플레이어도 0.5를 못 넘는 홀"은 반려.
4. **사람 검수 콘솔**: 목표물 미리보기 + QA 게이트 리포트(캘리브레이션 그래프 포함) +
   **직접 테스트 스윙 1회** UI. 승인/수정/반려. 검수자 인지 부하 최소화가 목적.
5. **홀 라이브러리**: 승인된 매니페스트를 태그(도메인/난이도/테마/비용)와 함께 적재.
   운영자는 여기서 골라 코스를 편성한다.

부수 효과: ③의 시뮬레이션 플레이어는 곧 **회귀 테스트**다. Judge 로직 변경 시 라이브러리 전체를
재캘리브레이션해 깨진 홀을 자동 탐지한다.

## 7. 미결 사항 (후속 논의)

- 홀 라이브러리·검수 콘솔의 제품 형태 (웹 어드민 vs 저장소 내 파일 + CLI)
- 도메인별 구현 우선순위와 단계 구분
- 점수→비거리 매핑의 도메인 간 형평성 검증 방법 (캘리브레이션 데이터 기반 보정 곡선 등)
- 시뮬레이션 플레이어의 약/중/강 프롬프트 실력 정의 방법
- SRE 티어 1의 임베딩 유사도 모델 선정과 한국어 진단문 처리

## 8. 비목표 (YAGNI)

- **영상 생성 홀**: 스윙당 생성 시간(수십 초~분)과 비용이 게임 루프를 깨뜨림.
- **라이브 인프라 시뮬레이션**: 실제 nginx/K8s를 띄우는 채점. 결정적 체커(§5.4)로 대체.
- **terraform plan 실행**: 실 provider 필요. HCL 구조 diff로 대체.
- **직군별 트랙 분리·팀 스크램블**: 개인전 혼합 코스로 확정.
- **채용/역량 평가 수준의 측정 신뢰도**: 사내 이벤트 맥락에 맞는 수준으로 조정.
